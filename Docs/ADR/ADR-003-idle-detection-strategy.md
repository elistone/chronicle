# ADR-003: Idle Detection Strategy

Status: Accepted

## Context

Chronicle needs to detect when the user is idle so that inactive periods are not incorrectly attributed to the last active application.

This ADR documents findings from a proof-of-concept spike validating idle detection on macOS before committing to a production implementation.

## Decision

Use `CGEventSource.secondsSinceLastEventType(.combinedSessionState, eventType:)` with event type `~UInt32(0)` (equivalent to `kCGAnyInputEventType`) to measure time since the last user input event, polled on a one-second `Timer`.

## Approach

`CGEventSource.secondsSinceLastEventType` is part of CoreGraphics. When called with:

- State: `.combinedSessionState` — reflects the current login session's HID event stream
- Event type: `CGEventType(rawValue: ~UInt32(0))` — the special "any input" sentinel that returns the minimum age across all event types (keyboard, mouse, trackpad, scroll wheel, stylus, etc.)

The return value is a `CFTimeInterval` (Double) representing seconds elapsed since the last input of any kind. This is polled every second with a `Timer`.

An idle threshold is applied to classify the user as Active or Idle.

## APIs Used

| API | Framework | Purpose |
|-----|-----------|---------|
| `CGEventSource.secondsSinceLastEventType(_:eventType:)` | CoreGraphics | Time since last user input |
| `Timer.scheduledTimer` | Foundation | 1-second polling interval |

## Accuracy

- Reflects the HID event stream — covers keyboard, mouse, trackpad, scroll, and stylus input.
- Resolution is approximately 1 second (limited by poll interval).
- The returned value is a monotonically increasing double that resets to zero on any input event.
- In practice, reads are instantaneous and do not themselves trigger activity.

## Permissions

No special entitlements or user-facing permission prompts are required. This API works inside the App Sandbox without any additional entitlements.

## Limitations

### Polling, not event-driven

The approach is poll-based (1-second interval). There is no notification or callback when the user transitions from Active to Idle. For Chronicle's use case (recording timelines at the minute scale) this is acceptable.

### Does not track media or touch input independently

The `~UInt32(0)` sentinel covers most HID input types but does not distinguish between input categories. Chronicle does not need this distinction, but it is worth noting.

### Screen lock / display sleep

When the screen locks or the display sleeps, `secondsSinceLastEventType` continues accumulating idle time. This is the desired behaviour — a locked screen should be recorded as an idle period.

### Remote control and accessibility input

Synthetic input generated by accessibility tools or remote control software (e.g. Screen Sharing) will reset the idle timer. This is consistent with the user being active via another means.

### Alternative: IOKit `IOHIDGetIdleTime`

IOKit provides `IOHIDGetIdleTime` via `IOHIDSystem`, which is the lower-level source that CoreGraphics ultimately reads. The CoreGraphics API is simpler to call from Swift and produces equivalent results. IOKit is recommended only if the CoreGraphics API becomes unavailable or insufficient.

### Alternative: `NSEvent.addGlobalMonitorForEvents`

Global event monitors deliver callbacks for every input event. This is event-driven but:

1. Requires Accessibility permission (`AXIsProcessTrustedWithOptions`) on macOS 10.15+.
2. Delivers raw events, requiring the application to maintain its own idle timer.

This approach is not recommended for idle detection given the permission requirement and added complexity.

## Consequences

- Idle detection is possible with no user permission prompt and no special entitlements.
- A 1-second poll interval is sufficient for timeline recording purposes.
- The CoreGraphics API is the simplest and most maintainable option available.
- A production implementation should make the idle threshold configurable (recommended default: 5 minutes).
- Idle periods should be recorded as a distinct event type in the activity timeline, not attributed to the last active application.

## Recommended Production Approach

1. Extract `IdleDetector` into `Features/ActivityTracking/` as a standalone observable class.
2. Inject an `idleThreshold: TimeInterval` parameter (default 300 seconds / 5 minutes).
3. Publish `idleSeconds: TimeInterval` and `isIdle: Bool` for consumers.
4. When transitioning from Active → Idle, record an idle-start event with timestamp.
5. When transitioning from Idle → Active, record an idle-end event with timestamp and duration.
6. The UI need not display raw idle seconds in production — status and duration are sufficient.
